Rapid Application Development Methodology

RAD

Key terms
  • Prototypes
  • Iteration

Model


The RAD model starts with:

  1. Determining the requirements-This is where documenting requirements gathered from questionnaires, observation and interviews. Acceptance tests are often paired with requirements.
  2. Build prototype-This is where a simpler design of the final product is made to show the customer what it should roughly look like or do.
  3. Evaluate prototype with customer-This is where the customer says what they like about the prototype and what they want changed.
  4. Is the prototype good enough?-This is where the Yes or No loop point is placed.
  5. If No to number 4 then it loops to improve the prototype and re-evaluate the prototype with the customer, then back to number 4
  6. If Yes to number 4 then Prototype becomes the fnal product.

    7. The advantages of RAD

    Helpful when requirements are unclear to the team.
    Should result in excellent usability for the customer
    Changing requirements can be accommodated.
    Encourages customer feedback in the project to get the best product.

    The disadvantages of RAD

    Efficient code is not prioritised.
    Demanding on customer’s time for feedback (the customer has to be accessible).

    When it might be used?

    RAD should be used only when a system can be modularized to be delivered in incremental manner.
    When the customer is unsure of their requirements.

    RAD Spiral Model

    Key terms
    • Prototypes
    • Iteration
    • Risk-analysis
    • Four quadrants

    Spiral Model


    The RAD Spiral model starts with:

    1. Quadrant 1-Determine objectives, alternatives and constraints.
    2. Quadrant 2-Evaluate alternatives, risks are identified and resolved. If risks are too hig, then the project may be stopped. A prototype is created.
    3. Quadrant 3-The prototype is developed.
    4. Quadrant 4-Planning takes place for the next phase.

    The advantages of the RAD Spiral Model

    Ideal for risky and complex projects.
    Changing requirements can be accommodated.
    Users see the system early.

    The disadvantages of the RAD Spiral Model

    Management is more complex and so can lead to further costs.
    Process is more complex
    Good risk analysts are expensive
    Efficient code is not prioritized

    When might it be used?

    Significant changes are expected in the product during the development cycle
    Customer is not sure of their requirements which is usually the case.
    When the risk is high
    RAD should be used only when a system can be modularized to be delivered in incremental manner.
